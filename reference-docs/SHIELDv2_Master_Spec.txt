================================================================================
0. WHAT IS SHIELD, IN ONE PARAGRAPH
================================================================================

SHIELD by Kentro is an operational cybersecurity assessment platform that
runs the full lifecycle of a Kentro consulting engagement - from client
self-signup, through service-specific intake questionnaires, through
consultant-driven analysis with AI assistance, all the way to executive
deliverables that clients can defend to their leadership and to their
auditors. The platform supports three service lines: Technical Debt,
Zero Trust (CISA ZTMM 2.0 and DoD Zero Trust Reference Architecture),
NIST CSF 2.0 (full Kentro CSF 2.0 Implementation Playbook methodology),
and MITRE ATT&CK coverage mapping. Every output is consultant-authored,
audit-defensible, and exportable as a polished PDF or XLSX deliverable.

1. LOCKED DECISIONS (these have been confirmed by Eugene)
================================================================================

  Tech stack:           Modern SPA - Next.js 14+ (App Router) + React +
                        TypeScript + Tailwind CSS + shadcn/ui components.
                        Backend: FastAPI (Python) for the API and worker
                        services. Database: PostgreSQL 16. Cache/queue:
                        Redis. Object storage: S3-compatible (AWS S3 in
                        GovCloud or Azure Blob in Gov).

  Hosting:              FedRAMP Moderate or High target. AWS GovCloud
                        (preferred) or Azure Government. No third-party
                        CDNs - everything self-hosted. No public-CDN
                        Tailwind/shadcn either; everything bundled.

  AI usage:             AI-augmented with mandatory PII redaction.
                        Provider is the developer's choice for v1 -
                        commercial Anthropic Claude, OpenAI, Azure
                        OpenAI, AWS Bedrock, Google Gemini, or any
                        other are all on the table. Eugene has
                        explicitly accepted the risk that the v1
                        LLM provider may NOT be FedRAMP-authorized
                        and that LLM data egress may therefore leave
                        the FedRAMP boundary. PII redaction (see
                        Section 12) is still mandatory as the
                        primary control. Provider can be tightened
                        to FedRAMP-only in a later release without
                        application changes (it's an environment
                        variable). AI proposes drafts; consultant
                        always has final editorial control.

  CSF methodology:      Full 10-step methodology from the CSF 2.0
                        Implementation Playbook. Tiered working profiles
                        (HIGH / MODERATE / LOW), 5-dimension scoring per
                        subcategory, weighted-floor roll-up to the
                        Enterprise Profile, IG metric alignment, gap
                        analysis with P1/P2/P3 prioritization, action
                        plan with POA&M linkage.

  Branding:             "SHIELD by Kentro." Wordmark "SHIELD" with a
                        small "by Kentro" subline. Header/footer carry
                        the Kentro mark.

  Roles:                Three roles - Admin (Kentro consultant), Reviewer
                        (Kentro auditor, read-only), Client (customer-
                        side user). The Client role includes executive-
                        style operational views by default - dashboards
                        lead with the findings a leader needs to defend
                        the engagement, not technical drill-down detail.
                        Technical detail is reachable but progressive.

  Tenancy:              Single-tenant - one SHIELD deployment per client
                        organization. Each deployment is its own FedRAMP
                        boundary. Kentro consultants connect into each
                        client's deployment to do their work (federated
                        OIDC against Kentro's IdP). No cross-tenant
                        analytics; cross-engagement insights happen in
                        Kentro's internal tooling, not in SHIELD.

  Things to keep from v1:
                        The login/signup screen visual language and the
                        registration flow (the conversational tone, the
                        step-by-step pace, the auto-save behavior). Take
                        these as your starting point and improve from
                        there.

  Authentication scope for v1:
                        MFA and email verification are DEFERRED for v1.
                        Client signup uses email + password only. Eugene
                        has explicitly accepted the risk. The auth code
                        path keeps MFA enrollment and email verification
                        as commented-out / disabled steps so they can be
                        re-enabled in a later release with a feature
                        flag rather than a code change. See section 4.5
                        and section 6.1 for the implementation pattern.

  Intake "I'm not sure" path:
                        The intake wizard's service-selection step
                        offers four service options PLUS a fifth option:
                        "I'm not sure - request a call with a consultant."
                        Picking the fifth option short-circuits the
                        wizard, collects a minimal "what's prompting
                        this" plus contact preference, and creates a
                        consultation request in the admin queue. See
                        section 6.2 step I1.

  Deliverable filename convention:
                        Every download link the platform generates uses
                        this filename format:
                          {Company_Name}_{Service_Name}{MMDDYY}.{ext}
                        Date is the deployment server's current date in
                        MMDDYY (e.g., 051826 for May 18, 2026). Spaces
                        in the company and service names become
                        underscores; non-alphanumeric characters are
                        stripped. Examples:
                          Nexus_Federal_Solutions_Tech_Debt_Review051826.pdf
                          Atlas_Defense_Solutions_CSF_2_0_Assessment051826.xlsx
                          Federal_Demo_Agency_Attack_Surface_Coverage051826.pdf
                        See section 16 for the full filename slugification
                        rules and edge cases.

2. THE FOUR SERVICES (what SHIELD does)
================================================================================

A client engagement with SHIELD always picks one or more services from
the four offered. The service choice drives the entire experience -
what questions they are asked at signup, what dashboards they see, what
deliverables they receive, what artifacts they are asked to provide.

Each service produces ONE primary executive deliverable plus supporting
artifacts. All deliverables are exportable as PDF (executive) and XLSX
(working).

--------------------------------------------------------------------------------
2.1  Service: Technical Debt Review
--------------------------------------------------------------------------------

  Purpose:        Identify overlapping, redundant, and shadow-IT
                  cybersecurity tooling. Recommend consolidation. Quantify
                  the savings opportunity.

  Primary input:  Client's software / capability inventory (Excel
                  spreadsheet or list of tools in use). SHIELD's AI helps
                  extract this from unstructured uploads (invoices,
                  contracts, screenshots) but the consultant approves the
                  final list.

  Methodology:    Categorize every tool (SIEM, EDR, NAC, etc.). Group by
                  function. Identify duplicates. Calculate total annual
                  spend per category. Highlight consolidation candidates
                  with estimated savings. Cross-reference with
                  organizational risk profile (do the duplicates exist
                  for redundancy reasons, or accidentally?). Map savings
                  to potential reinvestment in Zero Trust or ATT&CK gaps.

  Executive deliverable:
                  "Tech Debt Report" - 8-12 page PDF with:
                  - Executive summary (one page)
                  - Total spend, total identified savings, top 3
                    consolidation recommendations
                  - Category-by-category overlap table
                  - Cross-service bridge: "Here's where the savings could
                    fund Zero Trust gap closure or ATT&CK coverage"

  Working artifacts:
                  - Capability list (XLSX)
                  - Overlap analysis worksheet (XLSX)
                  - Final recommended consolidation plan (XLSX)

--------------------------------------------------------------------------------
2.2  Service: Zero Trust Assessment
--------------------------------------------------------------------------------

  Purpose:        Assess the client's Zero Trust maturity against either
                  the CISA ZTMM 2.0 or the DoD Zero Trust Reference
                  Architecture. Build a roadmap from current state to
                  target maturity.

  Sub-frameworks: Client picks ONE at intake:
                  - CISA Zero Trust Maturity Model 2.0
                      (federal civilian, state, local government,
                      critical infrastructure)
                  - DoD Zero Trust Reference Architecture
                      (DoD components, defense contractors, anyone with
                      CMMC or DoD ZT contract obligations)

  Primary input:  Service-specific Zero Trust questionnaire completed by
                  the client during intake. Questionnaire is open-ended
                  ("Walk me through how your users authenticate to this
                  system") with prompts. Plus optional supporting evidence
                  uploads.

  Methodology:    For each pillar (5 for CISA, 7 for DoD), score current
                  maturity and target maturity. Identify gaps. Sequence
                  remediation. Map gaps to specific MITRE ATT&CK
                  techniques that the gaps leave exposed (cross-service
                  bridge to Attack Surface mapping).

  Executive deliverable:
                  "Zero Trust Posture Report" - 10-15 page PDF with:
                  - Executive summary (one page, big number cards per
                    pillar)
                  - Pillar-by-pillar current vs. target with narrative
                  - Top 5 gaps and recommended remediation
                  - 12-month roadmap

  Working artifacts:
                  - Completed questionnaire (Word, mirrors the CSF
                    questionnaire format)
                  - Per-pillar scoring worksheet (XLSX)
                  - Gap and roadmap worksheet (XLSX)

--------------------------------------------------------------------------------
2.3  Service: NIST CSF 2.0 Assessment
--------------------------------------------------------------------------------

  Purpose:        Run a full CSF 2.0 assessment using the Kentro
                  methodology (see Section 8 of this doc for the full
                  implementation of the Playbook).

  Primary input:  System inventory categorized by FIPS 199 impact (HIGH,
                  MODERATE, LOW). Interview questionnaires per tier
                  (Step 1.1 / 1.2 / 1.3 from the existing toolkit, now
                  in-app forms). Supporting artifacts per the artifact
                  tracker.

  Methodology:    The Kentro CSF 2.0 Implementation Playbook, end-to-end.
                  Tier working profiles, 5-dimension scoring per
                  subcategory, weighted-floor roll-up to Enterprise
                  Profile, IG metric alignment, gap analysis, action
                  plan. See Section 8.

  Executive deliverable:
                  "CSF 2.0 Enterprise Profile and Action Plan" - 20-30
                  page PDF with:
                  - Executive summary (1-2 pages)
                  - Function-by-function maturity heatmap
                  - Tier-by-tier breakdown
                  - Gap analysis with priorities
                  - Action plan with owners and dates
                  - Methodology appendix

  Working artifacts:
                  - Three Tier Working Profiles (HIGH / MOD / LOW) - XLSX
                  - Enterprise Profile - XLSX (the "Step 3.2.1" file
                    shape)
                  - Gap Analysis with priorities - XLSX
                  - Action plan - XLSX or in-app
                  - Artifact tracker - in-app

--------------------------------------------------------------------------------
2.4  Service: MITRE ATT&CK Coverage Mapping
--------------------------------------------------------------------------------

  Purpose:        Map the client's existing security tooling against the
                  MITRE ATT&CK Enterprise matrix. Identify techniques the
                  client cannot detect, prevent, or respond to.

  Primary input:  Capability list from a Tech Debt engagement (re-used)
                  OR a freshly uploaded capability list. Optionally cross-
                  referenced with Zero Trust gap findings.

  Methodology:    For each ATT&CK technique, cross-walk the client's
                  capability list to determine detection / prevention /
                  response coverage. Score Covered / Partial / Uncovered.
                  Identify top blind spots. Tie blind spots back to Tech
                  Debt savings and Zero Trust gaps for a unified
                  recommendation.

  Executive deliverable:
                  "Attack Surface Coverage Report" - 8-15 page PDF with:
                  - Executive summary with three big number cards
                    (Covered / Partial / Uncovered)
                  - Top 5 blind spots in plain English
                  - Tactic-by-tactic coverage heatmap
                  - Recommendation paragraph that bridges to Tech Debt
                    savings and Zero Trust roadmap

  Working artifacts:
                  - Per-technique findings (XLSX)
                  - Methodology appendix (XLSX)

--------------------------------------------------------------------------------
2.5  The cross-service value loop
--------------------------------------------------------------------------------

When a client engages on more than one service, SHIELD synthesizes:

  Tech Debt savings           ->   redirected funding pool
  Zero Trust gaps             ->   prioritized capability investments
  ATT&CK uncovered techniques ->   detection / prevention / response shortlist
  CSF maturity gaps           ->   compliance framing for the spend

The Executive Summary on the client home shows: "Your team identified
$X in waste; we found Y critical gaps; here's the recommendation to
close them." This is the highest-value page in the entire platform.

3. ROLES AND PERMISSIONS
================================================================================

Three roles. Single-tenant deployment, so all users in a deployment
belong to one client organization (or are Kentro consultants connected
in).

--------------------------------------------------------------------------------
3.1  Admin (Kentro consultant)
--------------------------------------------------------------------------------

Who:        A Kentro consultant. There may be one or several active on
            any deployment depending on engagement size. Connected via
            federated OIDC against Kentro's IdP, not the deployment's
            client IdP.

Can:        Everything except permanently delete data (purge requires a
            second admin's approval). Create / archive projects, run
            analyses, score, approve, finalize deliverables, audit, view
            and edit any client artifact, write to the message thread,
            invite reviewers.

Cannot:     See data in other client deployments (single-tenant
            isolation). Modify the immutable audit log. Modify a
            deliverable that has been client-released without creating
            a new version.

--------------------------------------------------------------------------------
3.2  Reviewer (Kentro auditor / observer)
--------------------------------------------------------------------------------

Who:        A senior Kentro auditor or a third-party reviewer (e.g., the
            IG team during an FY review). Connected via federated OIDC.

Can:        Read everything in the deployment. Walk the audit chain on
            any subcategory or finding. Add review notes (separate from
            consultant's notes). Approve / decline an admin's final
            deliverable.

Cannot:     Modify scoring, edit questionnaire answers, run new analyses,
            send messages to the client, change any data. Read-only
            across the working surfaces with the exception of review
            notes.

--------------------------------------------------------------------------------
3.3  Client (customer-side user)
--------------------------------------------------------------------------------

Who:        Anyone at the client organization invited to or self-
            registered for this deployment. May be the CISO, the ISSO,
            a CFO, an analyst, an executive. The client side covers
            both POC and executive readers - the dashboards lead with
            executive-grade summaries and progressively reveal detail.

Can:        Complete intake. Upload documents. Answer questionnaires.
            Send messages to the consultant. View their org's
            deliverables. View their org's dashboards. Invite a
            colleague (with a self-imposed cap, e.g., 10 users per
            deployment for client side).

Cannot:     See or affect scoring math (read-only on results until the
            consultant releases). See consultant working notes or
            review notes. See raw AI output before consultant approval
            (no "Inspect output" JSON dumps). Reach the admin queue, the
            audit log, or the job/internals pages.

Client experience principles (this is critical):

  - Lead with the finding, not the methodology.
  - Show the number first, the explanation second, the drill-down third.
  - Never expose 5-dimension scoring, IG metric alignment, weighted-
    floor decision rules, or other methodology internals to the client.
    Those are consultant tools. The client sees "Maturity: Level 3 ->
    Target: Level 4" with one sentence of plain English.
  - Operational and executive views first. Technical and tactical
    detail behind a "Show me how this was calculated" link.
  - Reading time matters. Every primary client page should be
    skimmable in under 90 seconds.

4. ARCHITECTURE & TECH STACK
================================================================================

--------------------------------------------------------------------------------
4.1  Frontend
--------------------------------------------------------------------------------

  Framework:      Next.js 14+ (App Router, Server Components where
                  appropriate, Client Components for interactive surfaces)
  Language:       TypeScript (strict)
  Styling:        Tailwind CSS (self-hosted; no Tailwind CDN)
  UI components:  shadcn/ui (copied into the repo, NOT installed from
                  a CDN - per FedRAMP self-hosting requirements)
  Charts:         Recharts or Tremor (self-hosted)
  Forms:          React Hook Form + Zod
  State:          React Server Components for read paths;
                  TanStack Query for client-side mutations
  Auth (SPA side):
                  Next-Auth with OIDC adapter, or auth-js with custom
                  OIDC provider; calls Keycloak (or chosen IdP) for
                  tokens; SHIELD API validates JWTs.
  Routing:        File-based via Next.js App Router
  Accessibility:  WCAG 2.1 AA mandatory. All components keyboard
                  navigable. Color contrast verified at build time.

--------------------------------------------------------------------------------
4.2  Backend
--------------------------------------------------------------------------------

  Framework:      FastAPI (Python 3.12+)
  ORM:            SQLAlchemy 2.x with Alembic migrations
  Validation:     Pydantic v2
  Auth:           OIDC token validation; role mapping from IdP claims
                  to internal Role enum
  Worker:         Celery or RQ for async jobs (Redis-backed)
  Logging:        Structured JSON logs to stdout, shipped to the
                  deployment's SIEM/log aggregator
  Error handling: No stack traces ever returned to the user. Global
                  exception handler returns generic friendly error
                  with a correlation ID; admins can look up the real
                  error in logs by correlation ID.

--------------------------------------------------------------------------------
4.3  Data layer
--------------------------------------------------------------------------------

  Primary DB:     PostgreSQL 16
  Schema:         One schema per deployment (single-tenant); no row-level
                  client_id scoping is needed because the deployment IS
                  the boundary. Belt-and-suspenders: still keep a
                  client_id column on every row for audit clarity and in
                  case multi-tenancy ever becomes a requirement.
  Cache:          Redis (queue + ephemeral cache only; nothing
                  persistent)
  Object storage: S3-compatible (artifact files, generated PDFs/XLSX,
                  evidence uploads). Server-side encryption with
                  customer-managed keys (CMK).
  Audit:          Append-only audit_entries table with Postgres trigger
                  that rejects UPDATE/DELETE. Every state change writes
                  an audit row.

--------------------------------------------------------------------------------
4.4  AI / automated analysis
--------------------------------------------------------------------------------

  Provider:       Developer's choice for v1. Configured via environment
                  variable, NOT hardcoded - this is the single most
                  important architectural rule for this section, because
                  the provider choice WILL change in later releases.
                  Acceptable v1 options:
                    - Anthropic Claude API (commercial, fastest path)
                    - OpenAI API (commercial)
                    - Azure OpenAI (commercial or Government tier)
                    - AWS Bedrock (commercial or GovCloud when Claude
                      lands there)
                    - Google Gemini (commercial)
                    - Self-hosted open-weight model (e.g. Llama 3.1
                      70B / 405B on GPU) - heaviest to operate, but
                      keeps data inside the boundary
                  Eugene has accepted the risk that a v1 commercial
                  provider may NOT be FedRAMP-authorized and that
                  redacted payloads will egress to a non-FedRAMP
                  endpoint. This is a deliberate v1 trade-off for
                  velocity; the platform must be able to flip to a
                  FedRAMP-authorized provider via configuration alone
                  in a later release.

                  Required regardless of provider:
                    - All LLM API keys stored in a secrets manager,
                      never in code or in .env committed to source.
                    - Outbound LLM API calls go through a single
                      egress proxy / service module so the entire
                      surface can be audited and (in future) blocked
                      or redirected.
                    - Every LLM call logs the provider, model, prompt
                      version, input token count, output token count,
                      and the redaction summary to the llm_calls
                      table.

  Pattern:        Asynchronous job pipeline. Web request enqueues; worker
                  runs the analysis; client polls for status; admin
                  approves the AI's draft before it becomes part of the
                  deliverable.

  Redaction:      MANDATORY. Every payload that leaves the deployment
                  for the LLM API passes through the redactor:
                    - email addresses
                    - phone numbers
                    - personal names (best-effort, not perfect)
                    - street addresses
                    - the client's own organization name (replaced with
                      [CLIENT])
                    - common federal identifiers (SSN, EIN, CAGE, contract
                      numbers)
                    - signature blocks
                  The redaction layer records what was stripped in the
                  artifact's lineage so the audit log can prove redaction
                  ran. See Section 12.

  Output policy:  AI output is ALWAYS marked as a draft and never
                  rendered to a client until a consultant has explicitly
                  approved it. No "inspect raw JSON" boxes anywhere. AI
                  output is rendered as readable content (tables,
                  formatted text, charts) - never as <pre> blocks.

--------------------------------------------------------------------------------
4.5  Authentication and identity
--------------------------------------------------------------------------------

  IdP:            Keycloak (default; per-deployment). The client's IT
                  team can federate against their own IdP (Entra ID,
                  Okta, PIV/CAC) via standard OIDC connectors.
  Roles:          Mapped from IdP claims. Kentro consultants log in
                  through Kentro's IdP, federated against the deployment.

  MFA (DEFERRED for v1):
                  Email + password only for v1. Eugene has explicitly
                  accepted the risk. Implementation requirements:
                    - The Keycloak realm config MUST include the MFA
                      flow definition and the email-verification flow
                      definition - but both flows are DISABLED via
                      feature flag for v1.
                    - The codebase should reference MFA-required and
                      email-verified as token claims, but skip the
                      enforcement checks when the feature flag says
                      v1-auth-light is on.
                    - The /sign-in screen and /sign-up screen have
                      hooks where the MFA-enrollment and email-verify
                      step components would mount; for v1 those
                      components return null but the routes exist.
                    - The feature flag SHIELD_AUTH_REQUIRE_MFA defaults
                      to false in v1. Flipping it to true (and
                      SHIELD_AUTH_REQUIRE_EMAIL_VERIFY to true) in a
                      future release re-activates both flows with no
                      code change.
                    - Admin and Reviewer users connecting through
                      Kentro's IdP are still subject to whatever MFA
                      Kentro's IdP enforces; SHIELD does not loosen
                      THAT side of the auth (an admin's own IdP rules
                      apply at sign-in regardless of SHIELD's flag).

  Session:        Short-lived JWTs (15 min) + refresh token. Idle timeout
                  30 min. Forced re-auth daily.

  Onboarding (v1 flow with MFA + email-verify deferred):
                  1. Client visits /sign-up.
                  2. Provides display name, email, password (with
                     strength meter). Accepts Terms.
                  3. Account is created. No verification email; no MFA
                     enrollment. The user is signed in immediately
                     and lands on /intake.
                  4. The first user becomes the deployment's Primary
                     POC (a flag on User, not an admin promotion).
                  5. A Kentro consultant connects to the deployment
                     and begins work.

  Onboarding (future flow when v1.x ships with auth tightened):
                  ============================================================
                  The same /sign-up route adds steps S2 (email verify)
                  and S3 (MFA enroll) per section 6.1 before reaching
                  /intake. No data model changes - the User table
                  already has mfa_enrolled and email_verified_at
                  columns; v1 just leaves them NULL.
                  ============================================================

  Invitations:    Client users can invite colleagues via email-token
                  flow. Tokens are single-use, hashed in storage, 7-day
                  expiry. (NOTE: in v1 with email-verify off, the
                  invitation token itself is still emailed - that's
                  the only email the platform sends in v1.)

--------------------------------------------------------------------------------
5.3  Reviewer page hierarchy
--------------------------------------------------------------------------------

                                        engagements.
    /reviewer/walk/<service>           Audit walk surface for a service.

                                        -> gap -> remediation. For ZT,

    /reviewer/audit                    Read-only audit log.
    /reviewer/notes                    Reviewer's own notes (separate

8. CSF 2.0 METHODOLOGY IMPLEMENTATION (the Playbook in code)
================================================================================

The CSF 2.0 Implementation Playbook is in this folder. The Working
Profile, Reference Data, Enterprise Profile, Gap Analysis docs are the
authoritative source. The platform MUST implement these faithfully.

Key concepts to encode:

  Function:           5 functions (GOVERN, IDENTIFY, PROTECT, DETECT,
                      RESPOND, RECOVER). PLUS the 6th: GOVERN is the
                      CSF 2.0 addition.

  Category:           Sub-groupings within each function (e.g. GV.OC =
                      Organizational Context within GOVERN).

  Subcategory:        108 leaf items (GV.OC-01, GV.OC-02, ...). Every
                      scoring/scoping decision happens at this level.

  Tier profile:       Working Profile scoped to a FIPS 199 tier. One per
                      tier the client uses (HIGH and/or MODERATE and/or
                      LOW). All have the same 108 subcategory rows.

  Enterprise profile: Synthesized rollup of all tier profiles. One
                      per client engagement.

  5-dimension score:  Per subcategory per tier:
                          Governance               0-2
                          Policy & Process         0-2
                          Implementation           0-2
                          Monitoring & Measurement 0-2
                          Continuous Improvement   0-2
                      Total: 0-10. Mapped to Maturity Level 1-5:
                          0-2  -> Level 1 (Ad hoc)
                          3-5  -> Level 2 (Repeatable)
                          6-7  -> Level 3 (Defined)
                          8-9  -> Level 4 (Managed)
                          10   -> Level 5 (Optimized)

  Evidence rule:      If evidence cannot be produced, Implementation
                      cannot exceed 1 AND total maturity cannot exceed
                      Level 2. Encode as a validation rule.

  IG metric metadata: Per subcategory: which IG metric(s) it maps to,
                      Core vs. Supplemental, Primary vs. Supporting
                      alignment. This data ships in the Reference Data
                      tab of the existing XLSX. Import once into the
                      database.

  Weighted-floor roll-up (Step 8 / Gap Analysis Methodology):
                      The Enterprise score for each subcategory is
                      computed from the tier scores using these rules
                      in order (first matching rule wins):
                        1. All tiers same score -> that score
                        2. Primary alignment to a Core metric AND any
                           tier has a gap -> strict floor (lowest score)
                        3. HIGH tier is the lowest scorer -> HIGH's score
                        4. MODERATE is lowest, HIGH higher -> MOD's score
                        5. Only LOW is lowest AND subcategory is
                           Supporting-aligned or Supplemental -> HIGH/MOD
                           score with documented exception
                        6. Mixed otherwise -> lower score + reasoning

                      Rule 2 always overrides Rule 5.

  Gap detection:      Current < Target -> GAP. Otherwise OK.

  Gap prioritization: P1: Core metric + HIGH tier + multi-system gap
                      P2: Core metric OR HIGH tier
                      P3: All other gaps

  Action plan:        Per gap: owner, deadline, resources, success
                      criteria, group by initiative.

Export shapes:
        Tier Working Profile -> Step 2.1 / 2.2 / 2.3 XLSX (existing
                                shape from the reference toolkit)
        Enterprise Profile   -> Step 3.2.1 XLSX
        Gap Analysis        -> custom XLSX matching the methodology doc

Cross-reference:
        Every CSF subcategory should be cross-referenceable to a Zero
        Trust pillar AND a MITRE ATT&CK tactic, so the value loop can
        be computed at the deployment level.

                  Per CISA ZTMM 2.0 - between 5 and 9 functions per
                  pillar. Each function has a different focus
                  (authentication, identity stores, access management,
                  etc.).

  Scoring:        Per function, per pillar: current maturity + target
                  maturity. Aggregate to pillar score. Aggregate across
                  pillars for overall posture.

  Gap:            Current < Target per function.

  Cross-reference to CSF: each ZTMM function maps to one or more CSF
  subcategories. The value loop view shows both.

--------------------------------------------------------------------------------
9.2  DoD ZT Reference Architecture - structure
--------------------------------------------------------------------------------

  Pillars (7):    User, Device, Network/Environment, Application/Workload,
                  Data, Visibility & Analytics, Automation & Orchestration

  Phases:         Target phase (45 activities) / Advanced phase (107
                  activities) - 152 activities total

  Scoring:        Per activity: implemented (Yes / Partial / No / Planned)
                  Per pillar: % of Target activities implemented; %
                  of Advanced activities implemented

  Gap:            Any Target activity not Yes is a gap. Advanced
                  activities are next-phase recommendations.

  Cross-reference: each DoD ZT activity maps to one or more CSF

10. MITRE ATT&CK COVERAGE METHODOLOGY
================================================================================

Approach:

  Catalog:        Vendor the MITRE ATT&CK Enterprise matrix. Refresh
                  quarterly. ~600+ techniques and sub-techniques.

  Filter:         Out-of-the-box, score against a curated subset of
                  ~33-40 techniques most relevant to the client's
                  industry/posture. Configurable per engagement to
                  expand to the full matrix.

  Mapping:        For each technique:
                      Detection: which tools from the capability list
                                 can detect this technique
                      Prevention: which tools can prevent it
                      Response: which tools can respond to it
                  Tool-to-technique mapping starts from a curated DB
                  (vendor MITRE Engenuity ATT&CK Evaluations, NIST
                  CSF crosswalks, vendor detection libraries) and is
                  refined per engagement by the consultant.

  Coverage scoring:
                      Covered: detection AND prevention AND response
                                all present
                      Partial: one or two of the three present
                      Uncovered: none present

  Top blind spots: Plain English explanation per uncovered technique,
                   why it matters, recommended remediation.

  Cross-service:   Uncovered techniques bridge to Zero Trust gaps and
                   Tech Debt savings. The deliverable includes the
                   value-loop bridge paragraph.

12. AI USAGE AND PII REDACTION
================================================================================

Every LLM call MUST pass through the redactor before leaving the
deployment.

IMPORTANT CONTEXT FOR v1: Eugene has explicitly chosen to allow a
non-FedRAMP-authorized commercial LLM provider (Anthropic, OpenAI,
etc.) for v1. This makes the redaction layer the PRIMARY data
protection control - not a defense-in-depth layer. Treat the
redactor as a security boundary, not a cosmetic step. Specifically:

  - The redactor MUST have automated test coverage on every PII
    pattern (emails, phones, addresses, names, identifiers,
    signature blocks) with realistic adversarial test cases.
  - The redactor's output MUST be inspected by a human (the
    consultant) before a payload first reaches the LLM in any new
    engagement. The first call per service for each new client
    enters a "review" state where the consultant approves the
    redacted payload preview before it ships.
  - If the redactor encounters content it cannot confidently
    redact (e.g. an unstructured PDF in a non-English language),
    the system MUST refuse to send to the LLM and surface a
    consultant action: "We couldn't confirm this document is safe
    to redact. Review manually before automated analysis runs."
  - Every payload that egresses to a non-FedRAMP endpoint is
    audited (audit row, llm_calls row, redaction summary).
  - The deployment's threat model documents the residual risk and
    the redactor's role as primary mitigation.

--------------------------------------------------------------------------------
12.1  Redactor module (shield_ai/redact.py)
--------------------------------------------------------------------------------

  Function:           redact_for_ai(text) -> (cleaned_text, removed_dict)
  Function:           redact_payload(obj) -> (cleaned_obj, removed_dict)

  Strips (at minimum):
    - Email addresses
    - Phone numbers (US and international formats)
    - Personal names (best-effort, with a curated names dictionary)
    - Street addresses (with "Suite/Floor/PO Box" heuristics)
    - The client's own org name (replaced with [CLIENT])
    - SSNs, EINs, CAGE codes, contract numbers (regex)
    - Signature blocks ("Sincerely," "Regards," "Best,")

  Modes:              strict (everything above) / standard (above minus
                      org name and address) / off (engineering only,
                      never in production)

  Output:             removed_dict logs the count of each PII type
                      stripped. This goes into the artifact lineage
                      and the llm_calls table.

--------------------------------------------------------------------------------
12.2  Client disclosure
--------------------------------------------------------------------------------

On every workspace screen that runs automated analysis, a short
disclosure block:

  "Before we run the automated analysis, we automatically remove
   personal contact information - names, email addresses, phone
   numbers, and street addresses - from anything we send the analyzer.
   The names of products and vendors are kept so the analysis can
   work. You can see what was removed on each result's details page."

--------------------------------------------------------------------------------
12.3  AI output handling
--------------------------------------------------------------------------------

  - Render AI output as readable content (tables, formatted text,
    charts). Never as <pre> JSON dumps.
  - Always label as "Automated draft" until a consultant approves.
  - Consultant has explicit "Approve and use" action. Logged in audit.
  - Approved drafts become "Consultant approved" artifacts.
  - Released to client = consultant explicitly released. Until then,
    the client never sees the AI output or the draft.

14. WHAT NOT TO REPEAT FROM v1
================================================================================

These are the v1 mistakes the v2 build must avoid:

  1.  NEVER expose internal slug values to users. No "extraction_review",
      no "Zero_trust intake", no "ai_extraction". Every enum has a
      display-label map applied at the API layer or template render
      layer.

  2.  NEVER render raw JSON in user UI. AI output, lineage data, audit
      details all render as readable content. <pre>JSON</pre> is
      forbidden in any user-facing template.

  3.  NEVER expose stack traces to end users. Global exception handler
      returns generic message + correlation ID. Werkzeug-style debugger
      is disabled in any environment accessible to end users (CI
      enforcement: refuse to start if DEBUG=True and bound to non-
      localhost).

  4.  NEVER ask for the same data twice. User profile (title, phone)
      lives ON User; the Primary POC FK points to a User. Settings and
      intake share the same fields.

  5.  NEVER pre-populate a form with a single field's value for every
      column (the v1 "Acme Co everywhere" bug). Every save handler
      validates that field assignments come from the matching form key,
      not from any single object reference.

  6.  NEVER fire a service selection without creating ALL the
      corresponding project records. If client picks Zero Trust AND
      ATT&CK, the handler creates both projects (or both ServiceRequest
      records under the alternative pattern - pick one and stay
      consistent).

  7.  NEVER skip the admin notification on new client signup. Wire
      every high-priority event (new signup, intake complete, message
      from client, failed job) to the notification dispatcher.

  8.  NEVER leave the dashboard operationally empty. Every client home
      and every admin queue must answer "what should I do next?" without
      requiring drill-down.

  9.  NEVER use the native file input for document upload. Always
      drag-and-drop zone with visual feedback, file list, per-file
      progress, file type hints.

 10.  NEVER design questionnaires as a single massive scroll. Section
      tabs, progress bar, auto-save, resume capability are mandatory.

 11.  NEVER leave a queue or list without timestamps. "Waiting since"
      is operationally critical.

 12.  NEVER let the notification bell point to / (it must reach a
      working notifications page or be hidden until one exists).

15.5 DELIVERABLE FILENAME CONVENTION (applies to every download link)
================================================================================

Every file the platform makes available for download (PDF report,
XLSX export, evidence package, audit-walk export, etc.) MUST follow
this filename convention:

    {Company_Name}_{Service_Name}{MMDDYY}.{extension}

Where:
    {Company_Name}    The client organization's legal name,
                      slugified (see below).
    {Service_Name}    The service the deliverable is from,
                      slugified.
    {MMDDYY}          The deployment server's current date in
                      MMDDYY (month-day-year, two digits each).
                      Example: 051826 for May 18, 2026.
    {extension}       The file extension: pdf, xlsx, docx, zip, csv.

There is NO underscore between the Service_Name and the date - the
date concatenates directly to the service name. This matches the
example Eugene provided: Company_Name_Service_Name051826.

Slugification rules (apply identically to Company_Name and Service_Name):
    1.  Trim leading and trailing whitespace.
    2.  Replace any sequence of whitespace characters with a single
        underscore.
    3.  Strip any character that is not [A-Z], [a-z], [0-9], or
        underscore. (So "Acme, Inc." becomes "Acme_Inc", not
        "Acme,_Inc".)
    4.  Collapse runs of underscores to a single underscore.
    5.  Trim leading and trailing underscores.
    6.  Preserve case as entered (do NOT lowercase).
    7.  If the result is empty after slugification (edge case),
        substitute "Unknown".

Service name slugs (the canonical strings to use):
    Tech Debt Review                 ->   Tech_Debt_Review
    Zero Trust Assessment (CISA)     ->   Zero_Trust_Assessment_CISA
    Zero Trust Assessment (DoD)      ->   Zero_Trust_Assessment_DoD
    NIST CSF 2.0 Assessment          ->   NIST_CSF_2_0_Assessment
    MITRE ATT&CK Coverage            ->   MITRE_ATTACK_Coverage
    Consultation Request             ->   Consultation_Request
    Multi-Service Engagement Summary ->   Engagement_Summary

Examples (assuming a deployment server date of 2026-05-18):
    Nexus Federal Solutions Inc.    Tech Debt Review (PDF)
        -> Nexus_Federal_Solutions_Inc_Tech_Debt_Review051826.pdf

    Atlas Defense Solutions         NIST CSF 2.0 Assessment (XLSX)
        -> Atlas_Defense_Solutions_NIST_CSF_2_0_Assessment051826.xlsx

    Federal Demo Agency             MITRE ATT&CK Coverage (PDF)
        -> Federal_Demo_Agency_MITRE_ATTACK_Coverage051826.pdf

    Federal Demo Agency             Zero Trust Assessment - CISA (PDF)
        -> Federal_Demo_Agency_Zero_Trust_Assessment_CISA051826.pdf

Edge case rules:
    a) Versioning: if a deliverable is re-released on the same day,
       append "_v2", "_v3", etc. BEFORE the date:
            Nexus_Federal_Solutions_Inc_Tech_Debt_Review_v2_051826.pdf
       Wait - re-reading Eugene's spec: the format is exactly
       Company_Name_Service_Name051826 with the date glued to the
       service name. To preserve that, put the version AFTER the date
       but BEFORE the extension:
            Nexus_Federal_Solutions_Inc_Tech_Debt_Review051826_v2.pdf

    b) Bundles: a multi-service deliverable bundle (a single ZIP
       containing multiple service deliverables) uses
       "Engagement_Summary" as the service slug:
            Nexus_Federal_Solutions_Inc_Engagement_Summary051826.zip

    c) Working artifacts (admin-only intermediate files, not client-
       facing): prefix with "WORKING_" before the company name:
            WORKING_Nexus_Federal_Solutions_Inc_Tech_Debt_Review051826.xlsx
       These are tagged in the UI as internal-only and never surface
       on the client's Documents page.

    d) Time zone: the date is computed from the deployment server's
       local time at the moment the download URL is requested. NOT
       the time the underlying deliverable was finalized. This keeps
       the filename predictable from the user's perspective ("the file
       I download today has today's date in the name") even if the
       underlying deliverable was finalized last week.

    e) Filename length cap: if Company_Name + Service_Name exceeds
       200 characters total (rare but possible for long government
       agency names), truncate Company_Name to 150 characters and
       Service_Name to 50 characters before concatenation. Always
       preserve the full date and extension.

Implementation:
    - Centralize the filename-build function in one module (e.g.
      shieldv2_api/services/filename.py). Every download route calls
      this single function. NEVER inline filename construction in
      route handlers.
    - Unit test the slugifier against the company-name test cases in
      this folder's mock-data set: "Nexus Federal Solutions Inc.",
      "Atlas Defense Solutions, Inc.", "Federal Demo Agency", plus
      pathological cases like "Acme & Co., LLC", "U.S. Department of
      Anything", and "ABC123Org".

17. OPEN QUESTIONS / DECISIONS FOR EUGENE
================================================================================

These are second-tier decisions where the recommended option is listed
first. If any need to flip, talk to Eugene before coding.

Q1. Marketing landing page at /:
    Recommended: simple but polished one-pager describing SHIELD and
    Kentro, with sign-in / sign-up CTAs.
    Alternative: redirect / to /sign-in.

Q2. Self-registration vs. invite-only:
    Recommended: self-registration allowed (preserves the v1 process
    Eugene wants to keep). First user to register becomes the
    deployment's Primary POC. A Kentro consultant verifies and attaches.
    Alternative: invite-only - the Kentro consultant pre-creates the
    first client user.

Q3. Reviewer assignment model:
    Recommended: any admin in the deployment can attach a reviewer.
    Reviewer assignments are deployment-wide (a reviewer sees all
    services in this deployment, not service-by-service).
    Alternative: per-service reviewer assignment.

Q4. Deliverable approval flow:
    Recommended: admin marks deliverable "final"; reviewer (if any)
    approves; admin releases to client.
    Alternative: admin releases directly to client (no reviewer in the
    flow). Many engagements won't have a reviewer.

Q5. ATT&CK technique scope:
    Recommended: out-of-the-box, score against a curated 33-40 most-
    relevant techniques. Admin can expand to full matrix per
    engagement.
    Alternative: always full matrix (600+ items - heavy questionnaire).

Q6. AI provider for v1:
    Eugene's decision: developer's choice. Risk of non-FedRAMP egress
    explicitly accepted. PII redaction (Section 12) is the primary
    control.
    Recommended for v1: Anthropic Claude API (commercial). Fastest
    path; best output quality for the kinds of analytic prompts the
    platform will use; cleanest API for the redacted-payload pattern.
    Alternative for v1: OpenAI API or Azure OpenAI.
    Future tightening: configurable to FedRAMP-authorized provider
    (Azure OpenAI Gov, AWS Bedrock-with-Claude in GovCloud) via env
    variable when the deployment needs to assert FedRAMP boundary
    posture. No application changes required.

Q7. Languages and locale:
    Recommended: English only at v1.0. Build the platform i18n-aware
    (no hardcoded strings; locale-keyed message files) but ship with
    English locale only. Add languages in v1.x if needed.
